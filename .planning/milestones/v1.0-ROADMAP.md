# Roadmap: AIDX Website

## Overview

Refactor the static HTML prototype into a production-grade Next.js site mirroring the Ghostty website architecture. Three coarse phases: scaffold the project foundation, build the full component library and utilities, then wire up pages, docs content, assets, and verify the build.

## Phases

- [ ] **Phase 1: Project Scaffold** - Next.js setup, config, design tokens, fonts, and shared types
- [ ] **Phase 2: Component Library** - All UI components and lib/utility modules
- [ ] **Phase 3: Pages, Docs & Build** - Routing, docs content, public assets, and build verification

## Phase Details

### Phase 1: Project Scaffold
**Goal**: A runnable Next.js project with correct config, design tokens, fonts, and type foundations in place
**Depends on**: Nothing (first phase)
**Requirements**: SETUP-01, SETUP-02, SETUP-03, SETUP-04, SETUP-05, STYLE-01, STYLE-02, STYLE-03, FONT-01, FONT-02, LIB-09
**Success Criteria** (what must be TRUE):
  1. `npm install` and `npm run dev` succeed without errors
  2. `--brand-color`, gray scale, and syntax highlight CSS variables are defined and visible in browser DevTools
  3. Pretendard Std Variable and JetBrains Mono fonts load correctly in the browser
  4. TypeScript path alias `@/*` resolves to `./src/*` without compiler errors
  5. `npm run lint` passes on the scaffold with no errors
**Plans:** 2 plans

Plans:
- [x] 01-01-PLAN.md — Config files (package.json, tsconfig, biome, next.config) + npm install
- [x] 01-02-PLAN.md — Design tokens (globals.css), style types, font file, MDX plugin stubs

**UI hint**: yes

### Phase 2: Component Library
**Goal**: Every reusable UI component and lib utility is implemented, typed, and importable
**Depends on**: Phase 1
**Requirements**: COMP-01, COMP-02, COMP-03, COMP-04, COMP-05, COMP-06, COMP-07, COMP-08, COMP-09, COMP-10, COMP-11, COMP-12, COMP-13, COMP-14, COMP-15, COMP-16, COMP-17, COMP-18, LIB-01, LIB-02, LIB-03, LIB-04, LIB-05, LIB-06, LIB-07, LIB-08
**Success Criteria** (what must be TRUE):
  1. All 18 components import without TypeScript errors
  2. `AnimatedTerminal` renders and animates frames in the browser using `requestAnimationFrame`
  3. `Navbar` shows logo, nav links, and CTA on desktop; hamburger menu opens full-screen nav on mobile
  4. `Sidecar` tracks active heading as user scrolls a docs page
  5. Zustand docs store (`useDocsStore`) updates `headerIdsInView` as `JumplinkHeader` components enter/leave view
**Plans:** 3 plans

Plans:
- [x] 02-01-PLAN.md — Primitive components (Text, Link, Button, ButtonLinks, GridContainer, Terminal, AnimatedTerminal)
- [x] 02-02-PLAN.md — Docs UI components (Navbar, Footer, NavTree, Sidecar, Breadcrumbs, JumplinkHeader, CodeBlock, Callout, Blockquote, PathnameFilter, SectionWrapper, ScrollToTopButton)
- [x] 02-03-PLAN.md — Lib modules (config, navigation, page, store, MDX plugins, docs-mdx styles)

**UI hint**: yes

### Phase 3: Pages, Docs & Build
**Goal**: All pages are routed and rendered, docs content is live, assets are in place, and the static build passes
**Depends on**: Phase 2
**Requirements**: PAGE-01, PAGE-02, PAGE-03, PAGE-04, PAGE-05, PAGE-06, PAGE-07, PAGE-08, DOCS-01, DOCS-02, DOCS-03, ASSET-01, ASSET-02, ASSET-03, BUILD-01, BUILD-02, BUILD-03
**Success Criteria** (what must be TRUE):
  1. Homepage loads with animated terminal hero and CTA buttons
  2. Navigating to `/docs` renders the docs landing page with sidebar nav, sidecar TOC, and breadcrumbs
  3. At least one additional MDX doc page renders headings, code blocks, and callouts correctly
  4. Navigating to a non-existent route shows the 404 page
  5. `npm run build` completes without errors and produces a fully static output
**Plans:** 4 plans

Plans:
- [x] 03-01-PLAN.md — Root layout and homepage (layout, page, HomeContent, terminal-data, CSS modules)
- [x] 03-02-PLAN.md — Docs pages, 404 page, and MDX component map
- [x] 03-03-PLAN.md — Docs content files and public assets (nav.json, MDX pages, favicon, logo)
- [x] 03-04-PLAN.md — Build and lint verification + visual human check

**UI hint**: yes
